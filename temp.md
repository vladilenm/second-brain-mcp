# ТЗ: MVP приложения для работы с Second Brain через LLM и MCP

## 1. Цель проекта

Нужно создать MVP-приложение, которое подключается к существующему Second Brain пользователя.

Second Brain хранится в Obsidian Vault. Доступ к нему уже реализоан через MCP-сервевр.

Приложение должно стать удобным интерфейсом для работы с базой знаний через LLM:

* задавать вопросы по базе;
* искать и читать заметки;
* редактировать заметки через агента;
* создавать новые заметки;
* запускать готовые AI-процессы;
* сохранять историю действий;
* постепенно улучшать память агента.

Главная задача MVP — доказать, что пользователь может работать со своей базой знаний не вручную через Obsidian, а через агентский интерфейс.

---

## 2. Основной сценарий пользователя

Пользователь открывает приложение и видит дашборд.

Он может написать запрос:

> Найди всё, что у меня есть про AI Sprint, и собери из этого структуру MVP.

Система должна:

1. принять запрос пользователя;
2. обратиться к MCP-серверу;
3. найти релевантные заметки в Obsidian;
4. прочитать выбранные заметки;
5. собрать контекст для LLM;
6. сгенерировать ответ;
7. показать пользователю ответ и список использованных заметок;
8. предложить сохранить результат как новую заметку;
9. при необходимости обновить существующую заметку через подтверждаемый diff.

---

## 3. Ключевые разделы интерфейса

### 3.1. Чат

Главный экран приложения.

Функции:

* поле ввода запроса;
* история диалога;
* ответы модели;
* список источников из Obsidian;
* кнопки действий после ответа:

  * «Сохранить как заметку»;
  * «Обновить существующую заметку»;
  * «Запустить процесс»;
  * «Добавить в память агента»;
  * «Отметить ошибку».

Ответы должны явно показывать, на какие заметки они опирались.

---

### 3.2. Заметки

Экран для работы с Obsidian Vault.

Функции:

* поиск заметок;
* список найденных заметок;
* открытие заметки;
* просмотр markdown;
* редактирование markdown;
* кнопка «Обсудить с агентом»;
* кнопка «Улучшить заметку»;
* кнопка «Создать связанную заметку».

Важно: любые изменения заметки должны происходить только после подтверждения пользователя.

---

### 3.3. Редактирование заметки через агента

Сценарий:

1. Пользователь открывает заметку.
2. Пишет инструкцию, например:

   > Добавь раздел про marketplace процессов и сделай структуру понятнее.
3. Агент читает текущую версию заметки.
4. Агент предлагает новую версию.
5. Приложение показывает diff.
6. Пользователь нажимает:

   * «Применить»;
   * «Отменить»;
   * «Доработать».
7. После подтверждения файл обновляется через MCP.

Нельзя автоматически перезаписывать файлы без подтверждения.

---

### 3.4. Создание новой заметки

Сценарий:

1. Пользователь просит:

   > Создай новую заметку по этой идее в папке Projects/SecondBrain App.
2. Агент предлагает:

   * название файла;
   * путь;
   * содержимое заметки.
3. Пользователь подтверждает.
4. Заметка создаётся в Obsidian через MCP.

---

### 3.5. Процессы

Экран с готовыми AI-процессами.

В MVP нужно реализовать 3–5 процессов.

#### Процесс 1. Идея → структура проекта

На вход:

* хаотичное описание идеи;
* расшифровка голосового сообщения;
* текст заметки.

На выход:

* суть проекта;
* аудитория;
* боль;
* MVP;
* ключевые фичи;
* архитектура;
* следующий шаг.

---

#### Процесс 2. Заметки → Telegram-пост

На вход:

* заметка;
* несколько заметок;
* голосовая мысль;
* тема.

На выход:

* Telegram-пост в стиле пользователя;
* короткие абзацы;
* сохранение исходных мыслей;
* понятная структура.

---

#### Процесс 3. Идея → ТЗ для агента-разработчика

На вход:

* описание проекта или фичи.

На выход:

* цель;
* пользовательские сценарии;
* страницы;
* компоненты;
* API;
* структура данных;
* ограничения;
* критерии готовности.

---

#### Процесс 4. Разбор проекта

На вход:

* название проекта;
* выбранные заметки;
* описание состояния.

На выход:

* текущее состояние;
* что уже понятно;
* что неясно;
* риски;
* следующие шаги.

---

#### Процесс 5. Улучшение базы знаний

На вход:

* заметка;
* результат диалога;
* новый материал.

На выход:

* какие правила добавить;
* какие теги поставить;
* какие связанные заметки создать;
* что сохранить в память агента.

---

### 3.6. Память агента

Нужно создать отдельный раздел для управления памятью агента.

Память можно хранить в Obsidian в папке:

```txt
/AI-System
  role.md
  rules.md
  style.md
  projects.md
  mistakes.md
  examples.md
```

Функции:

* читать файлы памяти;
* редактировать файлы памяти;
* добавлять новые правила;
* сохранять хорошие примеры;
* сохранять ошибки;
* использовать эти файлы как контекст при каждом запросе.

---

### 3.7. История и логи

Приложение должно сохранять:

* запрос пользователя;
* ответ модели;
* использованные заметки;
* запущенный процесс;
* созданные файлы;
* изменённые файлы;
* ошибки;
* обратную связь пользователя.

Историю можно хранить в SQLite, Postgres или простом локальном JSON/файловом хранилище на этапе MVP.

## 5. Роли компонентов

### Frontend

Отвечает за:

* интерфейс чата;
* список заметок;
* редактор markdown;
* экран процессов;
* экран памяти агента;
* подтверждение изменений;
* отображение источников и diff.

---

### Backend

Отвечает за:

* общение с MCP;
* поиск заметок;
* чтение заметок;
* создание заметок;
* обновление заметок;
* сбор контекста для LLM;
* вызов LLM;
* запуск процессов;
* сохранение логов.

---

### MCP

Используется как слой доступа к Obsidian.

Нужные операции:

* поиск заметок;
* чтение заметки;
* создание заметки;
* обновление заметки;
* получение списка заметок;
* получение метаданных заметки.

Если каких-то методов в MCP нет, нужно сделать заглушки и явно указать, какие методы нужно добавить.

---

### LLM

Используется для:

* ответов по базе;
* анализа заметок;
* переписывания заметок;
* создания новых документов;
* запуска процессов;
* структурирования информации;
* улучшения памяти агента.

---

## 6. Основные API приложения

Нужно реализовать внутренние API-методы:

### POST /api/chat

Принимает:

* сообщение пользователя;
* выбранный режим;
* список выбранных заметок, если есть.

Делает:

* ищет контекст;
* читает заметки;
* собирает prompt;
* вызывает LLM;
* возвращает ответ и источники.

---

### GET /api/notes/search

Принимает:

* query.

Возвращает:

* список найденных заметок;
* путь;
* название;
* краткий фрагмент;
* дата изменения, если доступна.

---

### GET /api/notes/read

Принимает:

* path.

Возвращает:

* содержимое заметки.

---

### POST /api/notes/create

Принимает:

* path;
* content.

Создаёт новую заметку через MCP.

---

### POST /api/notes/update

Принимает:

* path;
* oldContent;
* newContent;
* userConfirmation.

Обновляет заметку только после подтверждения.

---

### POST /api/processes/run

Принимает:

* processId;
* input;
* selectedNotes.

Запускает выбранный AI-процесс.

---

### POST /api/memory/add

Принимает:

* тип памяти: rule / mistake / example / project;
* content.

Добавляет информацию в соответствующий файл памяти агента.

---

## 7. Режимы чата

В чате должно быть несколько режимов:

### Ask

Обычный вопрос по базе знаний.

Пример:

> Что я уже думал про AI Sprint?

---

### Edit

Редактирование выбранной заметки.

Пример:

> Улучши структуру этой заметки и добавь раздел про MVP.

---

### Create

Создание новой заметки.

Пример:

> Создай новую заметку из этого диалога.

---

### Process

Запуск готового процесса.

Пример:

> Преврати эту идею в ТЗ для разработчика.

---

## 8. Требования к безопасности данных

* Не перезаписывать заметки без подтверждения.
* Показывать diff перед изменением.
* Логировать все изменения.
* Хранить исходную версию заметки перед обновлением.
* Желательно иметь возможность отката изменения.
* API-ключи хранить только в env.
* Не отправлять в LLM весь vault целиком, только релевантные заметки.

---

## 9. Что НЕ входит в MVP

На первом этапе не нужно делать:

* полноценный SaaS;
* регистрацию пользователей;
* оплату;
* команды и workspace;
* публичный marketplace;
* сложные права доступа;
* мобильное приложение;
* сложную визуализацию графа заметок;
* автоматическую массовую переработку всей базы;
* многоагентную оркестрацию.

MVP должен быть личным инструментом для одного пользователя.

---

## 10. Критерии готовности MVP

MVP считается готовым, если пользователь может:

1. открыть приложение;
2. подключиться к своему MCP;
3. задать вопрос по базе;
4. получить ответ с источниками;
5. найти и открыть заметку;
6. попросить агента изменить заметку;
7. увидеть diff;
8. подтвердить изменение;
9. сохранить обновление в Obsidian;
10. создать новую заметку;
11. запустить хотя бы 3 готовых процесса;
12. сохранить результат процесса в Obsidian;
13. добавить ошибку или правило в память агента.

---

## 11. Главная формула проекта

Приложение должно работать по формуле:

```txt
User Request
→ Context Builder
→ MCP Search/Read
→ Agent Memory
→ LLM
→ Response
→ User Confirmation
→ MCP Write/Update
→ Logs
→ Improvement Loop
```

---

## 12. Главный результат MVP

На выходе должен получиться не просто чат с заметками, а рабочий прототип персонального AI-harness:

* с интерфейсом;
* с доступом к Obsidian;
* с LLM;
* с редактированием заметок;
* с процессами;
* с памятью;
* с циклом улучшения.

Этот MVP должен стать основой для будущего продукта: personal AI workspace + marketplace процессов.
